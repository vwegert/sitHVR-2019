  • great tool for certain applications

Namespaces

Customer (Y.../Z...)
  • freely available
  • local governance required
  • no protection against collisions
Partner (/FOOBAR/...)
  • enforce ownership
  • global protection enforced
  • distributed object editing prohibited

üî• No delivery into protected namespaces

Case in point: Namespaces. No open namespace concept like Java.

globally available customer namespaces; local governance: ensure consistent authorship. No provisions / conventions against creating object with same name.

example: encapsulate user master data, choose ZCL_USER_DATA - so can anybody else. perfectly legal, provided: disjunct system landscapes. prerequisite for implicit customer namespaces. No automated resolution of naming collisions - not possible(?)

ABAP commercial background, requirements: ownership, liability, consistency. Owner: full access, keep everyone else from producing objects w/in namespace. reserved partner namespaces.

protected namespaces, bound to object source system: inherently incompatible with DCVS; publish object in protected namespace to Git repository - OK. No import into just any other system.

target system belongs to producer - no delivery, local management - great application of abapGit allow customer import: distribute production key. impractical, renders protective function void

Bottom line: Don't use abapGit: great hammer - problem isn't nail-shaped

Change and Transport System (CTS)

Records changes to objects in a system and applies these changes to a target system.

‚§∑ diff / patch

Records changed objects in a system and transfers the changed objects to a target system.

‚§∑ flag / copy

CTS: most know it, (read sentence)

No. Close, but no cigar. Does not record the changes - only records changed objects. Does not apply changes, but transfers entire changed object. (sub-object, same principle)

Recording changes: "diff", applying changes: "patch" - not what CTS does: flags and copies objects in their entirety

CTS Design Considerations

  • Goal: Customer System Landscape Consistency
  • All systems known by unique SID
  • Consistent release/patch levels
  • Local source system for every customer object
  • Local governance of development

Step back: apparent flaws of CTS, but keep intended purpose in mind

goal: keep customer system landscapes consistent / manageable, large projects - design assumptions

knows every system, unique ID

consistent release and patch levels along transport routes (frozen zones)

distinct source system for every customer object - master object, changes

relies on local governance: ensure consistency of the developed objects

CTS neither intended nor designed as delivery tool, ship objects between disconnected system landscapes. What might go wrong, I hear you ask? Follow me into the Transport Horror Picture Show...

Modifications

Challenge

  • Disencouraged, but necessary (emergency corrections)
  • Possible with consumer key
  • CTS support limited: protection against overwriting modified objects accidentally

üî• No merge tooling: out of scope of (local!) CTS

modifications to your software in target systems.

sometimes necessary: no correction notes. depending on criticality of issue, local repair by modification required.

grant customers ability to modify software via consumer key - "light side". can not prevent dark side

CTS support: none. stop if encounters modified object; flag "ignore collision and import anyway". CTS will copy entire object, not patch - no merge. customer documentation, ressources to reapply modifications

looks like flaw of CTS, but: direct consequence of assumption that every customer object has local source system. modifications = advance corrections in emergency; apply correction in source system ASAP. CTS not designed to handle issues that occur during product delivery.

Solution

Full SPDD and SPAU support, including "Modifikationsabgleichstransporte"

(Sorry, no screenshots - our systems are clean...)

The solution provided by the AAK - actually, it isn't.

use the same "extended" import mechanism as SAP standard - full SPDD and SPAU functionality for granted. No provisions, customers use known tools.

support for "Modifikationsabgleichstransport", included in installation process in subsequent systems

Dependencies and Ordering

Challenge

  • no information about dependencies
  • no enforcement of import order
  • consistency across system landscape not guaranteed
  • very time-consuming to debug and fix
  • applies to both upgrades and component dependencies

üî• Consistent object delivery not guaranteed

common problem: CTS has limited safeguards against importing in wrong order, but no real dependency management, no check for missing, skipped, out-of-order transports

import transport in any order: up-, down-, crossgrading software or parts at will. Worse: correct order in dev system, order mixed up in subsequent systems (prod!)

Transportorderitis bad disorder: takes a lot of time to diagnose and treat

Dependencies and import order: seem like distinct issues, but just faces of same coin. Order of import during upgrade: each version has predecessor as dependency. Out of scope...

Ensuring consistent delivery of all objects under all circumstances using the CTS is hardly possible. A lot depends on the correct handling on the customer side, and - speaking from experience - a lot of things can and will go wrong there.

Solution

  • Specifications of dependencies in package attributes
  • Order of support packages enforded automatically
  • Mandatory check before installation

package registration: constraints. dependencies with or w/o version. upgrades: minimum previous version, prevents omitting of too many upgrades

customer can upload all packages and have installation tool perform multiple upgrades in one operation

support packages: SPAM enforces sequential import

checks mandatory during preparation of installation

dependency management on software component level, coarser than usual. AAK would support more fine-grained approach, but small number of large software components sufficient and reduces overhead

Removing Stuff

Challenges

Individual objects:

  • deletion supported
  • once only (w/o hacks)
  • even more import ordering issues

Entire software:

  • no deletion from original system
  • theoretically possible with separate system
  • huge manual effort, very error-prone

üßü Beware of zombies!

üî• No proven solution.

so far getting stuff into customer system - how get rid of it? two facets, depends on "stuff"

individual objects supposedly easy - CTS supports deletion of objects. But: not designed for general-purpose software delivery. export object deletion record once, no further transports. direct consequence: customers have to obey import order, may not skip transport, lest "zombie" objects occur (enhancement - class)

entire software package: no easy way. no deletion in customer system, can't assemble deletion transport from development system. second empty system required - might as well move to AAK

Solution

Individual objects:

  • SDC: include or ignore
  • decision required
  • Package attributes: prevent skipping of deletion

Entire software:

  • Deinstallable Add-Ons
  • customization of deletion process

Software Delivery Composer notices deleted objects, choice of including deletions: objects don't vanish unnoticed. in conjunction with package prerequisites, prevent undead objects

deinstallation of entire add-on: easy (guidelines); AAK version 5, deinstallable packages

process customizable, special preparations to keep customer system consistent after removal. functionally potentially complex, technically easy to handle
